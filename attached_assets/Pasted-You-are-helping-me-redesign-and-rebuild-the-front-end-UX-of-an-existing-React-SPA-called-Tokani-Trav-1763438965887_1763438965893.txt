You are helping me redesign and rebuild the front-end UX of an existing React SPA called Tokani TravelFlow (TTF) hosted on Replit.

TTF is a corporate travel approval system for Fiji and the Pacific. It is not a public booking site and not a self-service tool for travellers. It is an internal workflow tool used by:

Travel coordinators / PAs / admin staff (primary users)

Approval managers (line managers, executives)

Finance / budget holders

Travel Desk / corporate travel agency

Admins

Travellers do not log in or submit their own requests. Their details are selected by coordinators from a directory.

Your job:
Refactor and/or build the front-end UI components so the UX matches the spec below, without breaking existing business logic or API calls.

Use the existing stack (React + whatever styling is already used). Do not introduce heavy new frameworks. Use functional components and hooks.

1. HIGH-LEVEL UX GOALS

Primary user = coordinator/PA, not traveller.

Make data entry very fast: autocomplete, prefilled fields, multi-traveller selection, templates.

Use a multi-step wizard for creating a request instead of one giant form.

Managers see clean summary cards and big approve/reject buttons, not the whole long form.

Travel Desk has a queue view + simple action screen, not the full request form.

All key screens must be mobile-first and responsive (most approvals will be on mobile).

2. ROUTES / PAGES TO IMPLEMENT OR CLEAN UP

Please inspect the existing code and then:

/request/new – New Travel Request Wizard (coordinator flow)

/requests – List of requests (filterable by status, role, etc.)

/requests/:id – Request Detail + Timeline + Actions

/dashboard/coordinator – Coordinator “my work” dashboard

/dashboard/manager – Manager approvals dashboard

/travel-desk – Travel Desk queue and processing screen

If the exact routing is different, adapt the components to fit the current system, but keep the UX patterns described.

3. ROLES & PERMISSIONS (FRONT-END UX)

Assume we have roles like:

COORDINATOR

MANAGER

FINANCE

TRAVEL_DESK

ADMIN

On the front end:

Show different dashboards/navigation depending on the role.

Coordinators see tools to create and manage requests.

Managers see pending approvals and clear decision screens.

Travel Desk sees approved requests to quote/ticket.

Admins see configuration and reporting.

Do NOT build full auth, just respect whatever role state already exists.

4. /request/new – COORDINATOR WIZARD (CRITICAL)

Rebuild /request/new as a 4–5 step wizard optimized for coordinators/PAs creating requests for other people.

STEP 1 – Traveller(s)

Traveller selector:

Multi-select input that searches a “directory” of travellers.

When a traveller is selected:

Auto-fill Department.

Auto-fill Manager (line manager).

Fields:

Traveller(s) [multi-select]

Department (auto, editable)

Manager (auto, editable)

UX details:

Show “recent travellers” for speed.

A toggle for “Group travel request” when multiple travellers are added.

STEP 2 – Trip Details

Fields:

Trip Type: One-way / Return / Multi-city

Origin

Destination

Departure Date

Return Date

Are dates flexible? (Yes/No, optional ± days)

Special travel notes (textarea, optional – e.g. medical, seating, etc.)

Behaviour:

If Multi-city: show an editable list of sectors (e.g. NAN → SYD, SYD → POM).

Provide “Copy to all travellers” behaviour if multiple travellers.

STEP 3 – Cost & Policy

We only need high-level inputs, not exact fares.

Fields:

Cabin Requested: Economy (default), Premium, Business

Estimated Cost Band (dropdown):

< FJD 1,000

1,000 – 3,000

3,000 – 5,000

> 5,000

Funding / Project / Cost Code (text or select)

Attachments: allow upload of at least 1–3 files (conference invite, training letter, memo).

Policy indicator:

Show a small policy status pill:

“In Policy” (green)

“Out of Policy” (amber)

Show a short reason text, even if it’s a placeholder for now (e.g. “Cabin above allowed level”).

If you can’t implement full policy logic yet, still wire in a simple flag, so the UI is ready.

STEP 4 – Approval Routing

Show a simple “Approval Path” box:

e.g. Manager → Finance (if cost band high or out-of-policy) → Travel Desk.

Fields:

Primary approver: auto-filled manager, editable.

Optional additional approver(s): e.g., Finance, HR, Project Owner.

UX:

Coordinators must be able to override or add approvers within reason.

This step is about who will see the request, not all company rules hard-coded.

STEP 5 – Review & Submit

Show a clean summary card:

Traveller(s)

Dept / Manager

Purpose

Route and dates

Estimated cost band

Policy status

Funding code

Approval path

Buttons:

Submit Request

Save Draft

Duplicate (optionally later)

UX rules:

Display everything in a single, readable card, not a long form.

Make it mobile-friendly.

Wizard Technical Notes

Track current step in component state.

Use a progress indicator (e.g. Step 1 of 5).

Persist unsaved changes in local component state.

If easy, add simple autosave (e.g. localStorage) so form progress isn’t lost on refresh.

Ensure all critical views are usable on mobile screens.

Use clear headings, section dividers, and cards. Avoid clutter.

Keep labels and copy plain-English and non-technical:

Avoid GDS jargon.

Avoid showing internal codes like RBD, TST etc.
